# Typescript email ui
## Function
Create the web UI under `src/ui/` (React + Vite + TypeScript) and the Capacitor shell config (`capacitor.config.ts`); do not execute! The UI is the client of the Proxy pattern twice over: every screen obtains a `MailProvider` from the `ProviderRegistry` and a `MailStore` (see `typescript_mail_store.md`) from the composition root, and calls only interface methods ‚Äî it must compile and pass all tests with nothing but the in-memory `FakeProvider` and `FakeMailStore`, proving that adding IMAP or a different storage engine later requires zero UI changes. Synced mail persists in the store (SQLite on device), so reading and exact text search work offline. The app is fully functional for reading, tagging, searching, and sending mail. The Capacitor shell wraps the built web app for iOS/Android with native OAuth for Gmail sign-in; native platform setup commands (`npx cap add ios|android`) are documented in the README, not executed.

## User Stories

| actor | requirement | description |
|-------|-------------|-------------|
| human | a failed initial tag/capability load (e.g. `AUTH_REQUIRED` before Google sign-in, or offline) to replace the "Loading‚Ä¶" state with the normalized error message and a Retry button ‚Äî never an indefinite spinner | i see how to fix sign-in (or my connection) instead of an app stuck on "Loading‚Ä¶" |
| engineer | `npm run build` to require an explicit provider flag ‚Äî `npm run build -- --provider=<id>` (valid ids: `gmail`, `vite`; more platforms add one entry) ‚Äî routed through `scripts/build.mjs`; a missing or unknown `--provider` throws before any compilation starts, with an error naming the flag, showing the usage, and listing the known provider ids; the build compiles with `vite build --mode <id>` so Vite's `MODE` carries the choice ‚Äî no environment variable, nothing written to `.env.local` | a production bundle always states which mail platform it ships, chosen only by the flag |
| engineer | the flag parsing/validation to live in `scripts/providerFlag.mjs` as a pure function (`resolveProviderFlag(argv)`) unit-tested by vitest; the build script (`scripts/build.mjs` ‚Üí `runBuild.mjs`) to spawn `tsc -b` then `vite build --mode <id>` and write nothing else; **`npm run dev` to take the same `--provider` flag** via `scripts/dev.mjs` ‚Üí `runDev.mjs`, starting `vite --mode <id>` (default `gmail` when the flag is omitted; any other args forwarded to Vite); `vite` is the id of the in-memory demo provider | one flag ‚Äî `--provider` ‚Äî drives both build and dev (Vite's `--mode` is an internal detail users never type), and the orchestration is unit-tested without a real build or dev server |
| engineer | the composition root to select and register exactly one provider by branching **directly** on `import.meta.env.MODE` (the mode Vite sets from `--mode` and statically replaces at build time) ‚Äî `MODE === 'vite'` ‚Üí `FakeProvider` (the in-memory demo), every other mode (`gmail`, and dev/test/prod) ‚Üí `GmailProvider`; the branch reads `import.meta.env` itself so it is a build-time constant the bundler can fold; new platforms add one branch here plus one id in `scripts/providerFlag.mjs`; unit tests drive it with `vi.stubEnv('MODE', ‚Ä¶)` | provider choice is `--provider`-driven with no env var, defaults to `gmail`, and the flag is the only selector |
| engineer | `--provider=<id>` to be **1:1 with a single provider class**: a production build includes only the selected provider's implementation and dead-code-eliminates every other one ‚Äî a `--provider=gmail` bundle contains no `FakeProvider` code, a `--provider=vite` bundle contains no `GmailProvider` code ‚Äî which the constant `import.meta.env.MODE` branch above makes tree-shakeable (both classes are imported at the top of the composition root, but only the selected branch survives folding); verified by a build test that builds each mode and greps the bundle for the other provider's marker | each app-store bundle ships exactly the one platform it was built for ‚Äî no demo/test provider and no dead provider code rides along |
| engineer | a `vite` build to start with a seeded demo mailbox: the app entry loads `public/fixtures/fake-provider.json` at startup with `loadFakeFixtures` (only when `import.meta.env.MODE === 'vite'`) and hands it to the composition root via a `fakeFixtures` option, which seeds the registered `FakeProvider`; a missing or unreadable fixture file logs a warning and falls back to an empty mailbox instead of crashing | the demo build shows the full read flow out of the box, and anyone changes the demo mail by editing a JSON file ‚Äî no code edit, no rebuild |
| engineer | an end-to-end Playwright suite under `tests/e2e/` (run with `npx playwright test`; config starts the dev server with `--provider=vite`) driving a real browser through the core read flow over the seeded demo mailbox: the sidebar lists the demo tags, Refresh under a tag lists its threads, opening a thread shows its messages | user-facing behavior is verified in a real browser with no real account, per the repo's Playwright-first policy ‚Äî vitest keeps covering the contract layers |
