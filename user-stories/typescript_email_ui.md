# Typescript email ui
## Function
Create the web UI under `src/ui/` (React + Vite + TypeScript) and the Capacitor shell config (`capacitor.config.ts`); do not execute! The UI is the client of the Proxy pattern three times over: every screen obtains a `MailProvider` from the `ProviderRegistry`, a `MailIntelligence` (see `typescript_mail_intelligence.md`), and a `MailStore` (see `typescript_mail_store.md`) from the composition root, and calls only interface methods — it must compile and pass all tests with nothing but the in-memory `FakeProvider`, `FakeIntelligence`, and `FakeMailStore`, proving that adding IMAP, a different AI backend, or a different storage engine later requires zero UI changes. Synced mail persists in the store (SQLite on device), so reading and text search work offline. **AI is optional**: when configured, arriving threads are auto-classified and tagged, long threads open with a digest, compose starts from a draft, and search interprets natural language; without AI configured, the app remains fully functional for reading, tagging, and sending mail. The Capacitor shell wraps the built web app for iOS/Android with native OAuth for Gmail sign-in; native platform setup commands (`npx cap add ios|android`) are documented in the README, not executed.

## User Stories

| actor | requirement | description |
|-------|-------------|-------------|
| human | a failed initial tag/capability load (e.g. `AUTH_REQUIRED` before Google sign-in, or offline) to replace the "Loading…" state with the normalized error message and a Retry button — never an indefinite spinner | i see how to fix sign-in (or my connection) instead of an app stuck on "Loading…" |
| engineer | `npm run build` to require an explicit provider flag — `npm run build -- --provider=<id>` (valid ids: `gmail`, `fake`; more platforms add one entry) — routed through `scripts/build.mjs`; a missing or unknown `--provider` throws before any compilation starts, with an error naming the flag, showing the usage, and listing the known provider ids; the build writes the chosen provider id to `.env.local` so `npm run dev` uses it without manual re-entry | a production bundle always states which mail platform it ships; after building for a provider, dev mode automatically uses that provider |
| engineer | the flag parsing/validation to live in `scripts/providerFlag.mjs` as a pure function (`resolveProviderFlag(argv)`) unit-tested by vitest; the build script to spawn `tsc -b`, `vite build` (with `VITE_MAIL_PROVIDER` set to the chosen id), and finally write `VITE_MAIL_PROVIDER=<id>` to `.env.local` so the dev server picks it up; `fake` is a valid provider for tests and development | the exception paths are tested without running a real build, developers can build against the in-memory reference implementation, and dev mode respects the build choice automatically |
| engineer | the composition root to register the provider named by `VITE_MAIL_PROVIDER` from a known-provider map (`gmail` and `fake` today; new platforms add one entry), using the value from `.env.local` (set by the build script) in dev mode or from the build step in production, defaulting to `gmail` only when no `.env.local` exists, and throwing an error listing the known ids when set to an unknown value | the built bundle registers exactly the platform it was built for, dev mode respects the build choice via `.env.local`, and a bad value fails loudly at startup instead of silently shipping the wrong provider |
