# Typescript gmail proxy

Status: built — `src/providers/gmail/GmailProvider.ts`. This skill documents required behavior; the generalized contract every provider satisfies lives in `typescript_mail_provider.md`.

## Function
Create `src/providers/gmail/GmailProvider.ts`; do not execute! This class is the **first concrete Proxy** behind the `MailProvider` interface: a local surrogate for the remote Gmail server that fulfills every interface method by calling the Gmail API directly via REST. The UI never sees this class — it resolves it from the `ProviderRegistry` as a plain `MailProvider`. Authentication flows through native Capacitor OAuth (ASWebAuthenticationSession on iOS, Custom Tabs on Android, browser-based on web). All Gmail-specific knowledge in the app (API schema, error mapping, OAuth setup) lives in this one directory.

## User Stories

| actor | requirement | description |
|-------|-------------|-------------|
| engineer | `GmailProvider` to implement the `MailProvider` interface exactly, with a test asserting assignability | it registers into the `ProviderRegistry` as a drop-in and the UI cannot tell it from any other platform |
| engineer | tests to mock `fetch` (injected, never global) and never open a socket or trigger OAuth | all tests are deterministic and reproducible |
| engineer | test fixtures modeled on the Gmail REST API v1 shape (threads, messages, labels), using fake addresses | tests are realistic yet deterministic and touch no real account |
| engineer | the constructor to take `{getAccessToken, fetchFn?: typeof fetch}` where `getAccessToken: () => Promise<string>` is a required OAuth2 token supplier called per request (a static token wraps trivially; native platforms plug in Keychain/KeyStore-backed refresh without provider changes) | all API calls carry a fresh `Authorization: Bearer` header and token refresh stays outside the proxy |
| engineer | construction to perform no I/O; the first HTTP request (and first `getAccessToken` call) happens on the first interface method call (Proxy pattern: lazy initialization) | registering a Gmail account at startup costs nothing until the user opens it |
| engineer | interface methods mapped to Gmail REST endpoints: `listTags→GET /gmail/v1/users/me/labels`, `listThreads→GET /gmail/v1/users/me/threads?labelIds=…` **followed by** `GET /gmail/v1/users/me/threads/{id}?format=metadata` per listed thread (threads.list returns only ids and snippets, so summaries need the metadata fetch), `getThread→GET /gmail/v1/users/me/threads/{id}?format=full`, `getMessage→GET /gmail/v1/users/me/messages/{id}?format=full`, `send→POST /gmail/v1/users/me/messages/send`, `markRead/markUnread/addTag/removeTag→POST /gmail/v1/users/me/messages/{id}/modify` with `addLabelIds`/`removeLabelIds` (read state is the `UNREAD` label), `archive→POST /gmail/v1/users/me/threads/{id}/modify` removing `INBOX`, `trash→POST /gmail/v1/users/me/threads/{id}/trash` (Gmail forbids adding `TRASH` via modify) | the proxy stays a thin delegate whose every call is a documented Gmail API operation |
| engineer | tag semantics passed through untouched: Gmail labels arrive as the model's tags (`id→tagId`, `name→name`; `unreadCount` omitted in v1 — labels.list carries no counts), `labelIds` maps to `tagIds`, and the proxy never synthesizes folder/containment behavior on top of them | the tag-based model reaches the UI exactly as Gmail expresses it |
| engineer | Gmail wire JSON mapped field-for-field to the shared camelCase model types: `payload.headers` yields subject/from/to/cc/bcc (To/Cc/Bcc split on commas), body parts walked recursively for `text/plain`→`bodyPlain` and `text/html`→`bodyHtml` (base64url-decoded, absent parts omitted), `unread` = `labelIds` contains `UNREAD`, with a test per model type | the schema contract between Gmail API and app is enforced by tests, not by reading two documents |
| engineer | each message's `date` taken from Gmail's `internalDate` (a string of epoch milliseconds) converted with `Number()`, no header parsing | one canonical date per message, normalized exactly once |
| engineer | a thread summary derived from the metadata fetch: subject/from/snippet/date/tagIds from the newest message, `unread` true when any message carries `UNREAD`, `messageCount` from the message count | the thread list renders without fetching full bodies |
| engineer | `pageToken` passed through to Gmail verbatim and `nextPageToken` handed back opaquely; `pageSize` forwarded as `maxResults` | pagination stays platform-opaque through every layer |
| engineer | `send(draft)` to build an RFC 2822 message (To/Cc/Bcc/Subject + plain body), base64url-encode it, and POST it as `{raw}`, resolving with the created message's `id` as `messageId` | the compose flow can confirm the send and open the sent message |
| engineer | Gmail API error responses mapped to `MailProviderError`: `401/403` → `AUTH_REQUIRED`, `404` → `NOT_FOUND`, `429` → `RATE_LIMITED`, other HTTP 4xx/5xx → `PROVIDER_ERROR` (carrying Gmail's `error.message` when present); transport failures (fetch rejection, non-JSON body) and a rejecting `getAccessToken` thrown as `MailProviderError('NETWORK')` and `MailProviderError('AUTH_REQUIRED')` respectively | UI error handling stays provider-agnostic even when the API is unavailable |
| human | an `AUTH_REQUIRED` error to carry a message telling me to sign in with Google via the app's OAuth flow | i know how to re-authenticate instead of seeing a raw HTTP failure |
| engineer | `capabilities()` returning `{supportsTags: true, supportsSend: true, supportsArchive: true}` without any API call | the UI can enable Gmail affordances immediately |
| engineer | the composition root to register `GmailProvider` with a `getAccessToken` it receives via an optional `gmailAuth` option (production wires the platform's native OAuth flow; tests inject a stub); when no `gmailAuth` is given, the default supplier rejects, so every Gmail call surfaces `AUTH_REQUIRED` with the sign-in guidance instead of crashing | the app composes and starts before any account is signed in, and the sign-in prompt appears exactly where mail is first touched |
| engineer | when no `gmailAuth` is given but `VITE_GMAIL_ACCESS_TOKEN` is set, the composition root's default supplier to resolve that env token instead of rejecting | developers can run the app against a real mailbox with a token from the OAuth playground before the native OAuth plugin is wired, with zero code changes |
| engineer | no retry, caching, or offline-queue logic in v1, documented as a deliberate omission | the proxy stays a thin remote surrogate; resilience is a later, separately-tested layer |
| engineer | vitest as the test runner, sharing the fixture builders with the provider-interface tests where shapes overlap | one source of truth for what wire and model objects look like |
